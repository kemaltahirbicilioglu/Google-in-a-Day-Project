# Production Deployment Recommendations

## Scaling and Infrastructure

The current system runs on a single machine with file-based storage and Python threads. To move into production, the first step is replacing the filesystem with proper data stores: Redis for the crawl queue and visited-URL set (fast writes, TTL-based expiration for re-crawl policies), and Elasticsearch or PostgreSQL with full-text indexing for the inverted index (enabling richer ranking, fuzzy matching, and horizontal read scaling). The crawler workers should be extracted into independent processes managed by a task queue such as Celery or distributed across multiple nodes via Kubernetes Jobs, allowing the system to scale crawl throughput horizontally by adding machines. Each crawler node would own a region or URL partition, pulling work from the shared Redis queue and writing results to the centralized search index. The API layer (FastAPI) already runs asynchronously and can be scaled behind a load balancer (e.g., Nginx or an AWS ALB) with multiple uvicorn workers. For very large crawls, a URL frontier service should manage politeness per domain (separate rate limits per target host) and priority scheduling so high-value domains are crawled first.

## Monitoring, Reliability, and Compliance

Production deployments require structured observability across both the crawler and search paths. Crawler metrics -- pages crawled per minute, error rates by HTTP status, queue depth, back-pressure event frequency, and per-domain fetch latency -- should be exported to a time-series system like Prometheus and visualized in Grafana dashboards with alerting on anomalies (e.g., sustained queue saturation or elevated 5xx rates). Search metrics should track query latency (p50/p95/p99), result click-through rates, and zero-result queries to surface relevance gaps. On the reliability side, the system needs graceful failure handling: circuit breakers for unresponsive domains, dead-letter queues for repeatedly failing URLs, and automatic checkpointing so interrupted crawls resume without operator intervention. Data should be stored in compliance with relevant regulations (GDPR for EU-sourced content, robots.txt and crawl-delay directives for politeness). Finally, rate limiting and authentication should be added to the search API to prevent abuse, and all crawler traffic should be routed through rotating proxies with proper User-Agent identification to avoid being blocked and to respect target site terms of service.
